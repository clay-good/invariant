Isaac Sim/Lab still needs a GPU even headless (PhysX runs on GPU), but no rendering means it's dramatically faster and cheaper.


What you log per simulation step (not video, just data):


Joint states (positions, velocities) as numpy arrays or JSON
The command your cognitive layer proposed
Invariant's verdict (accept/reject + which checks fired)
The signed audit log entry
The random seed + episode config
That's it. Text and numbers. Tiny per-step. You save this to disk as compressed traces.




# Isaac Lab headless -- no display, no rendering
python train.py --headless --num_envs 2048 --task MyRobotTask
The key insight: save the random seeds. Isaac Lab simulations are deterministic given the same seed + config. If you save (seed, config) for every episode, you can perfectly reproduce any episode later by re-running it.


Phase 2: Record videos from replay (after the fact)
Two approaches, both work:


Option A: Deterministic replay (recommended)


# After bulk runs, pick interesting episodes
# Re-run them WITH rendering, same seed = same physics
python replay.py --seed 847291 --config episode_847291.json --record video.mp4
Isaac Lab supports --video --video_length 200 --video_interval 1 flags natively. You re-run the exact episode with rendering enabled. Same seed = identical physics = identical outcome. You just turn the camera on this time.


Option B: State trajectory replay
Save the full joint state trajectory (positions per timestep) during bulk runs. Then load that trajectory into a rendering session and set joint positions frame-by-frame:




# Pseudocode
trajectory = load("episode_847291_states.npz")
for frame in trajectory:
    robot.set_joint_positions(frame.positions)
    sim.step(render=True)  # rendering ON
    recorder.capture_frame()
This works even if you didn't save seeds. Slightly less faithful (no contact dynamics replay) but visually identical for demo purposes.


Option A is better because:
Zero extra storage during bulk runs (just save seed + config, a few bytes)
Perfect physics fidelity on replay
You can choose camera angles after the fact
You only render the 5-10 episodes you actually want videos of

Invariant is a cryptographic firewall between the AI brain and the robot body.


Today, when an LLM or RL policy controls a physical robot — a humanoid on a factory floor, an arm in a warehouse — there is nothing deterministic between the AI's probabilistic output and irreversible physical movement. The AI hallucinates a joint command, and the arm swings into a person. A prompt injection tells it to exceed its authority, and it does. After the incident, the logs are editable. Nobody can prove what happened.


Invariant is the hard gate. One Rust binary, sub-millisecond latency, sits between cognition and actuation. Every motor command must pass through it. It enforces:


Physics — 10 deterministic checks (joint limits, velocity, torque, workspace bounds, exclusion zones, self-collision, stability). If the command would break the robot or hurt a person, it's rejected.


Authority — Cryptographic chain-of-custody from the human operator all the way down to the individual joint command. The operator says "pick up the dishes." That intent gets scoped to specific joints with Ed25519 signatures at every hop. If the AI tries to move a joint it wasn't authorized to move — even due to prompt injection — it's cryptographically impossible. Not just blocked. Non-representable.


Signed actuation — The motor controller will only execute commands signed by Invariant's private key. Even if someone compromises everything between the firewall and the motor, unsigned commands are physically ignored.


Tamper-proof audit — Every decision is hash-chained and cryptographically signed. After an incident, you have non-repudiable proof of every command issued, who authorized it, and whether it passed or failed. This is the liability answer.

1. The Repo: Project "Verve"
Name Idea: Verve (Latin for vigor/spirit) or Synapse-OS. Let’s go with Verve.
Tagline: The Secure, Multimodal Nervous System for Agnostic Humanoid Intelligence.
Detailed Description
Verve is an open-source abstraction layer that sits between "The Body" (physical sensors or simulators) and "The Brain" (LLMs like Gemini, Claude, or DeepSeek).
Instead of writing code for a specific robot, you write code for The Humanoid Intent. Verve treats every sensor—whether it's a camera, a pressure sensor on a finger, or a microphone—as a secure telemetry stream.
The Architecture
* The Sensation Layer (Ingestion): A standardized API that accepts data from Isaac Sim, Gazebo, or real hardware. It normalizes disparate data types (LiDAR point clouds, RGB-D video, IMU vectors) into a "World State JSON."
* The Orchestrator (BYOK): A "Router" that decides which model handles what.
   * Gemini 1.5 Pro: Handles long-context visual spatial reasoning (e.g., "Where did I leave the keys?").
   * DeepSeek-R1: Handles complex logic and chain-of-thought planning for tasks.
   * Qwen-2.5-Coder: Generates the specific ROS2 Python scripts to move the robot's arm.
* The Kinetic Guardrail (The Security Engineer's Touch): A "Physical Firewall." Every command generated by the AI is intercepted here. It checks against physics-based constraints (e.g., "Don't swing the arm through a glass table") and security protocols (e.g., "Don't record audio in this geo-fenced room").
